# AIRL v0.2 — Execution Engine Consolidation

**Status:** Spec
**Date:** 2026-03-23
**Scope:** Remove IR VM and tree-walking interpreter. Consolidate to bytecode VM + Cranelift JIT as the single execution path. Compile contracts into bytecode. Retarget bootstrap compiler to emit bytecode. Achieve true bootstrap fixpoint.

---

## Problem Statement

AIRL v0.1 has four execution modes:

| Mode | Flag | Contracts? | Completeness |
|---|---|---|---|
| Tree-walking interpreter | (default) | Yes | Full — all features |
| IR VM | `--compiled` | No | Full minus contracts |
| Bytecode VM | `--bytecode` | No | Full minus contracts |
| Bytecode + Cranelift JIT | `--jit` (feature-gated) | No | Primitive-typed functions only |

This creates three critical problems:

1. **Safety and speed are mutually exclusive.** The only mode that checks contracts is the slowest. Switching to `--bytecode` for performance silently disables all verification. There is no `--bytecode --contracts` option. This directly undermines AIRL's core value proposition: mandatory contract verification.

2. **Semantic divergence across modes.** Each mode reimplements language semantics independently. Known divergences include: modulo-by-zero panics in bytecode but returns an error in the interpreter; edge cases around nil values, empty lists, and type coercion have not been rigorously proven equivalent. The 26 shared fixture tests exercise happy paths. Bugs hide in the gaps between modes.

3. **Quadratic maintenance cost.** Every new language feature must be implemented in four places: the interpreter's `eval_inner`, the IR compiler + IR VM, the bytecode compiler + bytecode VM, and potentially the JIT eligibility checker + Cranelift emitter. In practice, features get added to the interpreter first and the other modes lag or get incomplete implementations.

Additionally, the bootstrap compiler has an architectural problem: it runs on the interpreter (mode 1) and emits IR for the IR VM (mode 2). This is not true bootstrapping — the compiler does not target the runtime it executes on. The "fixpoint" test proves consistency between two different runtimes, not self-compilation in the traditional sense.

## Design: Consolidated Architecture

### Target State

```
Source → Parse → AST → Bytecode Compiler → Bytecode
                                              │
                                              ├─ Bytecode VM (baseline, always correct)
                                              │    ├─ Contract assertions compiled as opcodes
                                              │    ├─ All builtins dispatched
                                              │    └─ Self-TCO
                                              │
                                              └─ Cranelift JIT (hot functions, transparent)
                                                   ├─ Promoted based on call count
                                                   ├─ Contract assertions compiled to native checks
                                                   └─ Fallback to bytecode for ineligible functions
```

**One bytecode format. One set of semantics. Contracts always enforced. JIT is invisible.**

No user-facing execution mode flags. The runtime picks the execution strategy. `--jit` becomes the default behavior (when the `jit` feature is enabled), not an opt-in flag.

### What Gets Removed

| File | Purpose | Replacement |
|---|---|---|
| `crates/airl-runtime/src/eval.rs` | Tree-walking interpreter | Bytecode VM |
| `crates/airl-runtime/src/ir.rs` | IR instruction types | Bytecode instructions |
| `crates/airl-runtime/src/ir_vm.rs` | IR virtual machine | Bytecode VM |
| `crates/airl-runtime/src/ir_marshal.rs` | Value↔IR marshalling | Bytecode compiler handles this |
| `run-ir` builtin | Executes IR from bootstrap compiler | `run-bytecode` or direct execution |
| `--compiled` CLI flag | Selects IR VM mode | Removed |
| `--bytecode` CLI flag | Selects bytecode mode | Bytecode becomes the default |

The `--jit` flag is retained during transition but should eventually become the default when the `jit` feature is enabled, with no flag needed.

### What Gets Added

1. **Contract opcodes in bytecode.** New instructions:
   - `AssertRequires { clause_reg, msg_reg }` — check precondition, halt with contract violation on false
   - `AssertEnsures { clause_reg, msg_reg }` — check postcondition after function body
   - `AssertInvariant { clause_reg, msg_reg }` — check invariant after body, before postcondition
   - These compile from the `:requires`, `:ensures`, and `:invariant` clauses in function definitions

2. **Builtin completeness in bytecode VM.** Verify and fix dispatch for all ~48 builtins used by bootstrap code. Known areas requiring verification:
   - Map builtins: `map-get`, `map-set`, `map-keys`, `map-values`, `map-has`, `map-size`, `map-new`, `map-from`, `map-remove`
   - String builtins: `char-at`, `substring`, `chars`, `split`, `join`, `contains`, `starts-with`, `ends-with`, `index-of`, `trim`, `to-upper`, `to-lower`, `replace`
   - I/O: `print`, `read-file`, `write-file`
   - List primitives: `head`, `tail`, `empty?`, `cons`
   - Type operations: `type-of`, `int-to-float`, `float-to-int`

3. **Bootstrap compiler retargeted to bytecode.** `compiler.airl` emits bytecode instructions instead of IR nodes. The bytecode instruction set is a better compilation target than the IR — it has explicit registers, is flatter, and is closer to what a real backend produces.

4. **True bootstrap fixpoint test.**
   ```
   Stage 0: compiler.airl → bytecode (compiled by Rust-side bytecode_compiler.rs)
   Stage 1: Stage 0 runs compiler.airl → produces bytecode-B
   Stage 2: bytecode-B runs compiler.airl → produces bytecode-C
   Fixpoint: bytecode-B == bytecode-C
   ```

## Implementation Plan

### Phase 1: Verify Bytecode VM Completeness

**Goal:** Run all bootstrap tests on the bytecode VM. Identify and fix gaps.

**Steps:**
1. Run each bootstrap test with `--bytecode` flag:
   ```bash
   cargo run -- run --bytecode bootstrap/lexer_test.airl
   cargo run -- run --bytecode bootstrap/parser_test.airl
   cargo run -- run --bytecode bootstrap/eval_test.airl
   cargo run -- run --bytecode bootstrap/types_test.airl
   cargo run -- run --bytecode bootstrap/compiler_test.airl
   cargo run -- run --bytecode bootstrap/equivalence_test.airl
   ```
2. Record all failures — categorize as: missing builtin, incorrect behavior, crash/panic, or unsupported feature.
3. Fix each gap in the bytecode compiler (`bytecode_compiler.rs`) and bytecode VM (`bytecode_vm.rs`).
4. Run all fixture tests (`tests/fixtures/valid/`) on bytecode to confirm no regressions.

**Acceptance criteria:** All 26 valid fixture tests and all bootstrap unit tests pass on `--bytecode`.

**Estimated scope:** Small-to-medium. The bytecode VM already handles closures, pattern matching, function refs, and self-TCO. Gaps are likely limited to missing builtin dispatches (map/string operations) and possibly edge cases in pattern matching on deeply nested variants.

### Phase 2: Compile Contracts to Bytecode

**Goal:** `:requires`, `:ensures`, and `:invariant` clauses produce bytecode assertion instructions. Contracts are always checked, regardless of execution mode.

**Steps:**
1. Add new bytecode opcodes:
   ```rust
   // In crates/airl-runtime/src/bytecode.rs
   AssertRequires { clause: u16, message: u16 },  // registers holding bool + error string
   AssertEnsures { clause: u16, message: u16 },
   AssertInvariant { clause: u16, message: u16 },
   ```
2. In `bytecode_compiler.rs`, compile contract clauses:
   - Each `:requires` clause compiles to: evaluate expression → `AssertRequires`
   - Each `:ensures` clause compiles to: (after body) bind `result` → evaluate expression → `AssertEnsures`
   - Each `:invariant` clause compiles to: (after body, before ensures) evaluate → `AssertInvariant`
3. In `bytecode_vm.rs`, implement assertion opcodes:
   - On true: no-op, continue execution
   - On false: return `RuntimeError::ContractViolation` with clause text and captured variable bindings (matching current interpreter behavior)
4. Run contract error fixtures (`tests/fixtures/contract_errors/`) on bytecode — they must produce the same errors.

**Acceptance criteria:** All contract error fixtures pass on bytecode. All valid fixtures still pass. No performance regression >5% on non-contract code (assertion opcodes on the hot path should be a single branch).

**Design note on contract compilation:** The current interpreter evaluates contract clauses by calling `self.eval()` on the clause expression within the function's scope. The bytecode equivalent is to compile the clause expression as additional bytecode after the function body, with `result` bound to the body's return register. The clause expression uses the same compilation path as any other expression — no special cases needed except the `result` binding and the assertion opcode at the end.

### Phase 3: Retarget Bootstrap Compiler to Bytecode

**Goal:** `compiler.airl` emits bytecode instructions instead of IR nodes. The compiled output runs on the bytecode VM.

**Steps:**
1. Define a bytecode serialization format in AIRL. The IR compiler currently uses tagged lists like `(IRCall "fname" [args...])`. The bytecode equivalent would be:
   ```lisp
   (BCFunc "name" num-registers [
     (LoadConst 0 42)
     (Add 2 0 1)
     (Call 3 "other-fn" [0 1 2])
     (Return 3)
   ])
   ```
2. Rewrite `compiler.airl` emission functions to produce bytecode instructions instead of IR nodes. The compilation logic (variable binding, scope management, recursion) stays largely the same — only the output format changes.
3. Replace the `run-ir` builtin with a `run-bytecode` builtin (or repurpose `run-ir` to accept bytecode format).
4. Update `bootstrap/equivalence_test.airl` to compare bytecode execution results against expected values.

**Acceptance criteria:** `compiler.airl` compiles a test suite of programs to bytecode. Executing the bytecode produces identical results to running the same programs directly.

**Note:** This is the largest phase. The IR instruction set has ~18 node types. The bytecode instruction set has ~34 opcodes. The mapping is roughly 1:N (each IR node may expand to multiple bytecode instructions, particularly for pattern matching and function calls). The compiler will grow but the logic is mechanical — it's a lowering pass, not a redesign.

### Phase 4: Bootstrap Fixpoint on Bytecode

**Goal:** Prove the bootstrap compiler achieves true fixpoint: the compiled compiler compiles itself identically.

**Steps:**
1. Stage 0: Compile `compiler.airl` to bytecode using the Rust-side `bytecode_compiler.rs`. Call this BC-0.
2. Stage 1: Run BC-0 on the bytecode VM with `compiler.airl` as input. It produces bytecode output BC-1.
3. Stage 2: Run BC-1 on the bytecode VM with `compiler.airl` as input. It produces bytecode output BC-2.
4. Assert: BC-1 == BC-2 (byte-for-byte or instruction-for-instruction after normalization).

**Acceptance criteria:** Fixpoint test passes deterministically on release builds. BC-1 == BC-2.

**Design note on determinism:** The current IR fixpoint uses string comparison via `ir-to-string`. The bytecode fixpoint should use a similar bytecode serialization (`bytecode-to-string`) for human-readable diffing, plus raw instruction comparison for the actual assertion. Register allocation must be deterministic (the current linear allocator is deterministic, so this should hold).

### Phase 5: Remove Dead Code

**Goal:** Delete the interpreter and IR VM. Clean up all references.

**Steps:**
1. Remove files:
   - `crates/airl-runtime/src/eval.rs` (tree-walking interpreter)
   - `crates/airl-runtime/src/ir.rs` (IR instruction types)
   - `crates/airl-runtime/src/ir_vm.rs` (IR virtual machine)
   - `crates/airl-runtime/src/ir_marshal.rs` (IR marshalling)
2. Remove from `crates/airl-runtime/src/lib.rs`: module declarations for `eval`, `ir`, `ir_vm`, `ir_marshal`.
3. Remove `run-ir` from builtins registry in `builtins.rs`.
4. Remove `--compiled` flag from CLI argument parsing in `crates/airl-driver/src/main.rs`.
5. Remove `run_source_with_mode()` interpreted path and `run_source_compiled()` in `pipeline.rs`. Replace with single `run_source()` that always uses bytecode.
6. Make `--bytecode` the default (no flag needed). Retain `--jit` during transition.
7. Remove `Interpreter` struct and all references. The `BytecodeVm` becomes the sole runtime.
8. Update `CLAUDE.md`, `README.md`, and all design docs to reflect the consolidated architecture.
9. Remove dead bootstrap tests: `bootstrap/equivalence_test.airl` (compared interpreter vs IR VM — both gone), `bootstrap/fixpoint_test.airl` (replaced by bytecode fixpoint in Phase 4).

**Acceptance criteria:** `cargo build` succeeds. `cargo test --workspace` passes (test count will decrease — that's expected). No references to `eval.rs`, `ir_vm.rs`, or `--compiled` remain outside of git history and design docs.

### Phase 6: Widen JIT Eligibility (Future)

**Goal:** More functions become JIT-eligible over time, without user action.

This phase is not part of the v0.2 consolidation but is the natural follow-on. Currently the JIT only handles primitive-typed functions (no lists, closures, or variants). Widening eligibility is how the bytecode VM gets faster without adding new execution modes:

1. **List operations via runtime helper calls.** JIT-compiled code calls into Rust helper functions for `head`, `tail`, `cons`, `empty?` instead of interpreting them. The function stays JIT-compiled; list operations are native calls.
2. **Variant construction/matching via tagged pointers.** Variants become `(tag, payload)` pairs in the JIT — tag is an integer, payload is a pointer to heap-allocated data. Pattern matching becomes a switch on the tag.
3. **Closure calls via indirect call.** The JIT emits an indirect call through a function pointer stored in the closure object.

Each expansion makes more real-world functions JIT-eligible without any user-facing change. The bytecode VM remains the fallback for anything the JIT can't handle.

## Impact on Existing Code

### Bootstrap Compiler

| Component | Impact |
|---|---|
| `bootstrap/lexer.airl` | No change to source. Runs on bytecode VM instead of interpreter. Should be faster (bytecode is 4-10x faster than interpreted for heavy recursion). |
| `bootstrap/parser.airl` | Same as lexer. |
| `bootstrap/eval.airl` | Same. |
| `bootstrap/typecheck.airl` | Same. |
| `bootstrap/compiler.airl` | **Rewritten** in Phase 3 to emit bytecode instead of IR. Core compilation logic (scoping, variable resolution, recursion handling) stays the same; output format changes. |
| `bootstrap/equivalence_test.airl` | **Replaced** — no longer compares interpreter vs IR VM. New test compares bytecode execution results against expected values. |
| `bootstrap/fixpoint_test.airl` | **Replaced** with bytecode fixpoint (Phase 4). |
| All other `*_test.airl` files | Run on bytecode VM. Should pass without changes if Phase 1 is complete. |

### Standard Library

No changes to stdlib source (`.airl` files). The stdlib is loaded and evaluated via bytecode instead of the interpreter. The `eval_prelude()` function in `pipeline.rs` is replaced by `compile_and_run_prelude()` that bytecode-compiles each stdlib module and executes it on the bytecode VM.

### Fixture Tests

All fixture tests in `tests/fixtures/` run on the bytecode VM. Contract error fixtures require Phase 2 (contracts in bytecode) to produce correct errors. The fixture runner in `fixtures.rs` is updated to use `BytecodeVm` instead of `Interpreter`.

### CLI

| Before | After |
|---|---|
| `cargo run -- run file.airl` (interpreter) | `cargo run -- run file.airl` (bytecode, contracts enforced) |
| `cargo run -- run --compiled file.airl` (IR VM) | Removed |
| `cargo run -- run --bytecode file.airl` (bytecode, no contracts) | `cargo run -- run file.airl` (bytecode, contracts enforced) |
| `cargo run --features jit -- run --jit file.airl` | `cargo run --features jit -- run file.airl` (JIT automatic) |

### Performance Expectations

| Workload | v0.1 (interpreted) | v0.2 (bytecode) | v0.2 (bytecode+JIT) |
|---|---|---|---|
| Bootstrap lexer (self-lex) | ~56s | ~10-15s (est.) | ~10-15s (closures not yet JIT-eligible) |
| fib(30) | ~42s | ~4.6s | ~10ms |
| Fixture test suite | ~5s | ~2s (est.) | ~1s (est.) |
| Contract overhead | 0 (always on) | ~5-10% vs no-contract bytecode | ~5-10% |

The key tradeoff: v0.2's bytecode is faster than v0.1's interpreter for everything, but the bytecode now includes contract checking that v0.1's bytecode skipped. Net result should still be faster than v0.1's interpreted mode (the only v0.1 mode that checked contracts).

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Bytecode VM missing builtins for bootstrap code | High | Medium | Phase 1 is specifically designed to find and fix these. Run bootstrap tests first. |
| Contract compilation changes bytecode layout, breaks JIT | Medium | Medium | JIT compilation runs after contract opcodes are inserted. JIT eligibility checker skips assertion opcodes (they're VM-only). |
| Bootstrap compiler rewrite (Phase 3) introduces new bugs | Medium | High | Rewrite incrementally: one compilation function at a time, with test coverage at each step. Keep the IR compiler available on a branch until fixpoint is proven. |
| Performance regression from contract opcodes in hot loops | Low | Low | Assertion opcodes are a single conditional branch (well-predicted). Measured overhead should be <5%. Profile if higher. |
| Fixpoint test doesn't converge | Low | High | If register allocation is deterministic (it is — linear allocator), fixpoint should hold. Debug with bytecode-to-string diff if it doesn't. |

## Sequencing and Dependencies

```
Phase 1 (bytecode completeness)
    │
    ├─→ Phase 2 (contract opcodes)      ── independent of Phase 3
    │       │
    │       └─→ Phase 5 (remove dead code) ── requires Phase 2 + Phase 4
    │
    └─→ Phase 3 (retarget compiler)
            │
            └─→ Phase 4 (fixpoint test)
                    │
                    └─→ Phase 5 (remove dead code)
```

Phases 2 and 3 can proceed in parallel after Phase 1. Phase 5 requires both Phase 2 and Phase 4. Phase 6 is future work with no dependency on Phase 5.

## Definition of Done

AIRL v0.2 is complete when:

1. `cargo run -- run file.airl` executes on the bytecode VM with contracts enforced
2. All 517+ existing tests pass (some will be replaced, total count may change)
3. All bootstrap tests run on bytecode VM
4. Bootstrap compiler emits bytecode and achieves true fixpoint (BC-1 == BC-2)
5. `eval.rs`, `ir.rs`, `ir_vm.rs`, `ir_marshal.rs` are deleted
6. No `--compiled` flag exists. No `--bytecode` flag needed (it's the default).
7. Contract error fixtures produce correct errors on bytecode
8. `CLAUDE.md` and `README.md` updated to reflect single-execution-path architecture
