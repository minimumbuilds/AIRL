# AIRL v0.2 Performance Results

**Date:** 2026-03-23
**Binary:** release build with `--features jit`
**Methodology:** 3 runs per task, median taken
**Cached artifact:** `benchmarks/artifacts/v0.2/airl-driver-release-jit` (34MB)

## What Changed in v0.2

- Bytecode VM is now the default execution path (was tree-walking interpreter)
- Contracts (`:requires`/`:ensures`/`:invariant`) are compiled to bytecode assertion opcodes
- Contracts are always enforced — no mode skips them
- `--compiled` and `--bytecode` flags removed
- `--jit` flag retained for Cranelift JIT on eligible functions

### Summary

| Metric | Default | JIT | Python |
|--------|---------|-----|--------|
| Correct | 24/25 | 24/25 | 25/25 |
| Total time | 150ms | 203ms | 1,439ms |
| Avg per task | 6ms | 8ms | 57ms |
| vs Python | **0.10x** | 0.14x | 1.00x |

**AIRL's bytecode VM is ~10x faster than Python on startup-dominated tasks.** This is primarily startup time — AIRL's native binary starts in ~4ms vs Python's ~40ms import overhead.

**JIT is slower than default on these tasks** because JIT compilation overhead (~2ms per function analysis) exceeds the execution time of trivial programs. JIT only pays off on heavy computation.

## Heavy Workload: fib(30)

| Mode | Time | vs Python |
|------|------|-----------|
| AIRL default (bytecode) | 5,809ms | 23x slower |
| AIRL JIT | 5,943ms | 24x slower |
| Python | 251ms | baseline |

### Why JIT Doesn't Help fib(30)

**All functions with contracts are JIT-ineligible.** The v0.2 contract opcodes (`AssertRequires`, `AssertEnsures`, `AssertInvariant`) are in the JIT's disqualifying opcode list. Since AIRL mandates non-empty contracts on every function, **no user function can be JIT-compiled in v0.2.**

This is a critical regression from v0.1, where the bytecode path had no contracts and the JIT could compile pure-integer functions like `fib`. The v0.1 JIT achieved 10ms on fib(30) — a 460x speedup over bytecode. v0.2's safety guarantee (contracts always enforced) eliminated the JIT's primary use case.

### The Tradeoff

| | v0.1 | v0.2 |
|---|---|---|
| fib(30) bytecode | 4,605ms | 5,809ms |
| fib(30) JIT | 10ms | 5,943ms (JIT ineligible) |
| Contracts enforced | Only in interpreter (166s) | Always (5.8s) |
| Safety guarantee | Opt-in (slow mode only) | Always-on |

v0.2 chose safety over speed. The correct fix is to make the JIT contract-aware (compile assertion opcodes to native conditional branches) rather than disqualifying contracted functions. This is future work.

## Next Steps

1. **Make JIT contract-aware** — compile `AssertRequires`/`AssertEnsures` to native conditional branches in Cranelift instead of disqualifying contracted functions. This would restore the 460x JIT speedup while preserving safety.
2. **Profile bytecode VM** — the 5.8s fib(30) is 23x slower than Python. The bytecode VM's main loop likely has optimization opportunities (fewer clones, register access optimization).
