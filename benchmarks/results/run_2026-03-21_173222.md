```lisp
(defn safe-divide
  :sig [(a : i64) (b : i64) -> Result[i64, String]]
  :requires [(valid a)]
  :ensures [(valid result)]
  :body (if (= b 0)
          (Err "division by zero")
          (Ok (/ a b))))

(let (r1 : Result[i64, String] (safe-divide 10 3))
  (match r1
    (Ok v)  (print "safe-divide(10, 3) =" v)
    (Err e) (print "Error:" e)))

(let (r2 : Result[i64, String] (safe-divide 10 0))
  (match r2
    (Ok v)  (print "safe-divide(10, 0) =" v)
    (Err e) (print "Error:" e)))

`★ Insight ─────────────────────────────────────`
- **`(valid a)` as a minimal guard**: When there's no meaningful precondition to express, `(valid x)` always returns `true` — it satisfies AIRL's requirement that every function has at least one contract clause without over-constraining.
- **Why not `(requires [(not (= b 0))])`?** We *could* require `b != 0` as a precondition, but then calling with `b = 0` would be a *contract violation* (a crash). By handling zero in the body and returning `(Err ...)`, we make the function total — it handles all valid inputs gracefully.
```

#### Task 02: Fibonacci

```lisp
(defn fibonacci
  :sig [(n : i64) -> i64]
  :requires [(>= n 0) (<= n 30)]
  :ensures [(>= result 0)]
  :body
    (if (= n 0) 0
      (if (= n 1) 1
        (+ (fibonacci (- n 1)) (fibonacci (- n 2))))))

(print (fibonacci 10))

`★ Insight ─────────────────────────────────────`
1. **Contracts as documentation + enforcement**: The `:requires` clauses `(>= n 0)` and `(<= n 30)` are checked at runtime in `checked` mode and can be verified statically by Z3 at call sites. This is more powerful than a simple `assert` — it's part of the function's type-level interface.
2. **Nested `if` instead of `match`**: AIRL's `match` works on variant types like `Result`, not integer ranges, so nested `if` is the idiomatic way to handle base cases. Both branches are always required — there's no `cond` or `when` form.
3. **Why cap at 30**: Naive recursive Fibonacci is O(2^n). At n=30 this is ~1 billion operations — feasible but slow. The contract prevents accidentally calling with n=40+ which would hang. AIRL's recursion depth limit of 50,000 would also kick in well before that.
```

```python
def fibonacci(n):
    assert isinstance(n, int), "n must be an integer"
    assert n >= 0, "n must be non-negative"
    assert n <= 30, "n must be no greater than 30"

    if n == 0:
        return 0
    if n == 1:
        return 1

    a, b = 0, 1
    for _ in range(2, n + 1):
        a, b = b, a + b

    assert b > 0, "result must be positive for n >= 1"
    return b


result = fibonacci(10)
assert result == 55, f"Expected 55, got {result}"
print(result)

`★ Insight ─────────────────────────────────────`
- **Iterative over recursive**: Uses O(n) time and O(1) space instead of naive recursion's O(2^n). The n≤30 cap would make recursion feasible, but iterative is still better practice.
- **Contract pattern**: The precondition asserts mirror AIRL's `:requires` clauses, and the postcondition assert (`b > 0`) mirrors `:ensures` — validating output invariants before returning.
```
